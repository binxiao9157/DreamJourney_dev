## 1. 总体判断

当前工程没有偏离“记忆沉淀 + 克制陪伴 + 家庭关怀 + 隐私分层”的大方向，但和《阶段一.docx》的完整产品目标相比，仍存在明显差距。

一句话概括：

> 当前工程更像“勿忘 App 的阶段一工程底座”，而《阶段一.docx》定义的是“数字哀伤辅导工具 + 空巢老人适老终端 + 家庭账户闭环 + B2B2C 数字疗法服务”的完整阶段蓝图。

按不同口径估算：

| 口径 | 当前达成度 | 说明 |
| --- | ---: | --- |
| A 端 App 核心链路 | 65% - 70% | 记忆档案馆、KBLite、时空信箱、数字人、关怀看板、后端 metadata 能力已有雏形。 |
| 阶段一完整产品规划 | 45% - 55% | B 端适老终端、声纹模型自动生成、临床级危机干预、专家导流、机构服务闭环基本未实现。 |
| 真机真实验收 | 35% - 45% | 代码和脚本较多，但真实素材、真实账号、真实后端、连续对话和跨设备证据仍不足。 |

## 3. 主要差异总览

| 差异类型 | 具体表现 | 影响 |
| --- | --- | --- |
| 产品定位差异 | 文档强调“急性哀伤期数字疗法/心理辅助干预”；当前实现更偏“家族记忆 App + 数字人陪伴 + 档案沉淀”。 | 真实验收时可能缺少哀伤辅导专属流程、风险干预和告别仪式感。 |
| 建库闭环差异 | 文档要求“1 张旧照片 + 3-5 段微信语音截图 + 口头禅/性格描述，几分钟生成声纹模型和人格 Prompt”；当前只有素材管理、分析、声纹样本状态和 speakerId 绑定。 | 距离用户可感知的“一键快速建库”还有明显差距。 |
| 声音克隆差异 | 文档要求以逝者声音回信/陪伴；当前主要是火山 TTS 音色、speakerId 或系统 TTS fallback。 | 数字哀伤产品的核心冲击力不足，但这也需要强伦理边界。 |
| 时空信箱体验差异 | 文档要求信纸式 UI、发光万寿菊、跨越时空动效、几分钟/几小时后带语音回信；当前是功能型信箱和安全文本回声。 | 仪式感不足，用户心理感受和产品记忆点弱。 |
| 危机干预差异 | 文档要求后台 NLP 实时扫描极端词、立刻停止扮演、弹 24 小时热线、通知紧急联系人；当前是本地规则/guard 骨架和危机 UI。 | 安全能力还达不到数字疗法方向的产品承诺。 |
| 适老终端差异 | 文档要求脱离复杂 App 的智能音箱或单按钮小程序；当前只有 iOS App 内的一键语音/关怀数据链路。 | “一个大脑，两副面孔”只完成了大脑和 A 端，B 端基本未成型。 |
| 潜行建库差异 | 文档要求老人日常聊天中持续沉淀多年私有知识库；当前有对话沉淀，但没有长期设备端、云端生命周期、家庭账户绑定闭环。 | 终极商业故事还没有工程闭环。 |
| 商业化差异 | 文档包含保险、医院、EAP、专家转介、家庭安心报订阅、康养服务撮合、数字传记收费；当前没有机构端、支付、专家服务、运营后台。 | 当前只适合产品验证，不适合商业闭环验证。 |

## 4. 逐项详细对比

### 4.1 产品定位与目标人群

| 《阶段一.docx》目标 | 当前实现 | 差距 | 优先级 |
| --- | --- | --- | --- |
| 核心定位是“数字哀伤辅导工具”，服务急性哀伤期失亲者。 | 当前 App 已有数字人、信箱、档案、安全边界，但交互默认更像家族回忆采集和陪伴对话。 | 缺少“刚失去亲人”的专属 onboarding、哀伤阶段识别、告别任务、情绪容纳流程。 | P0 |
| 补充人群是极度孤独空巢老人。 | 已有长辈关怀看板、语音对话、规则化健康信号。 | 缺少适老独立入口、极简一键设备、老人身份下的免建库流程。 | P1 |
| 用户处于情绪撕裂、PTSD/MDD 风险状态。 | SafetyMonitor、SafetyGuard、危机页存在。 | 没有临床风险分层、危机联系人、热线配置、人工专家转接。 | P0/P1 |

判断：当前工程的人群覆盖“泛家庭记忆用户”更宽，但对“急性哀伤期”的产品化刻画不够深。

### 4.2 极低门槛“记忆提取器”

| 《阶段一.docx》目标 | 当前实现 | 差距 | 优先级 |
| --- | --- | --- | --- |
| 上传 1 张旧照片。 | 记忆档案馆支持照片导入、后端图片分析、KBLite 入库。 | 真实照片分析还需真机和后端验收；UI 还偏素材列表，不够“保险柜/相册集”。 | P0 |
| 上传 3-5 段微信语音截图。 | 支持截图、OCR、语音样本导入、语音转写、声纹样本状态。 | “微信语音截图 -> 语音内容/人物/声纹素材”的自动链路不完整；截图不能真正提取微信语音音频。 | P0 |

### 4.3 时空信箱

| 《阶段一.docx》目标 | 当前实现 | 差距 | 优先级 |
| --- | --- | --- | --- |
| 放弃微信式即时聊天，采用投递与回声。 | 已有信件、延迟投递、本地通知、边界回声。 | 已对齐主方向。 | P1 |
| UI 像精美信纸，背景有发光万寿菊。 | 当前是功能型信箱界面。 | 仪式感、视觉隐喻和哀伤疗愈氛围不足。 | P2 |

## 5. 目前最明显的方向偏差

### 5.1 从“哀伤疗愈”偏到了“家族记忆工具”

当前工程在足迹、档案、图谱、亲友、回忆录上投入较多，已经很像一个家族记忆产品。
但《阶段一.docx》的第一性目标是急性哀伤期用户的情绪安全和好好告别。

纠正方向：

- 首页和核心路径要围绕“我想说的话”“我还没说完的话”“今天先把这份想念安放好”。
- 记忆档案馆不是普通资料库，而是为了建立安全、克制、可追踪的陪伴素材。
- 数字人不应自由聊天，而应服务告别、倾诉、边界和回到现实。

### 5.2 从“克制延迟”偏到了“实时数字人聊天”

文档明确反对微信式秒回，强调投递与回声。
当前数字人实时对话能力很强，但也最容易制造“复活式沉迷”。

纠正方向：

- 实时数字人要被限制在“引导讲述/建库/情绪树洞”，而不是无限模拟逝者聊天。
- 与逝者相关的回复应更多走时空信箱的延迟回声。
- 实时对话中如果没有授权记忆，必须明确“不知道”，不能编造亲密细节。

### 5.3 安全系统还不够产品化

当前有 SafetyMonitor 和 guard 骨架，但《阶段一.docx》要求的是明确的生命红线干预闭环。

纠正方向：

- 增加用户可配置紧急联系人。
- 增加热线和地区资源配置。
- 危机触发时停止数字人角色扮演。
- 危机事件不进入普通记忆沉淀。
- 形成可验收的危机场景脚本。

### 5.4 B 端适老终端基本缺席

当前阶段一开发大量集中在 App。
但文档的“一个大脑，两副面孔”要求 B 端是产品闭环的一半。

纠正方向：

- 阶段一可以先做“App 内适老极简模式”作为 B 端最小替代。
- 后续再拆小程序/音箱。
- 先把老人端体验压缩成：一个按钮、一个状态、一个退出方式、无复杂 Tab。

## 6. 建议的后续开发优先级

### P0：必须补齐，否则和《阶段一.docx》核心不一致

1. 哀伤辅导 onboarding：
   - 用户选择“失去的人是谁”“离别大概多久”“今天想做什么：写信/建档/说说话”。
   - 不做医疗诊断，但用于控制对话频率、边界文案和引导策略。

2. 记忆提取器最小闭环：
   - 一个页面完成旧照片、语音/截图、口头禅、性格描述上传。
   - 生成“建库完成度”和“可用于陪伴的素材清单”。
   - 明确哪些素材可生成、哪些只本机保存。

3. 时空信箱产品化：
   - 延迟投递是主体验，而不是附属功能。
   - 回信必须有边界声明。
   - 如果启用 TTS，只能用已授权 voiceType/speakerId，并明确“基于素材合成”。

4. 生命红线闭环：
   - 危机词触发后停止角色扮演。
   - 弹热线/紧急联系人。
   - 不把危机文本沉淀为普通记忆。

5. 数字人记忆约束：
   - 有证据才回答。
   - 没证据时承认不知道并引导用户补充档案。
   - 日志和 UI 能显示“引用了哪些已授权记忆”。

### P1：阶段一真机验收应尽快覆盖

1. 长辈关怀跨设备验收：
   - 邀请、接受、撤回。
   - 子女只看脱敏周报。
   - 后端 403 验证。

2. App 内适老极简模式：
   - 一个大麦克风。
   - 无复杂菜单。
   - 对话结束自动沉淀 KBLite。
   - 子女端能看到脱敏信号。

3. 档案馆 UI 贴近“保险柜/相册集”：
   - 按人物/素材类型/可生成状态组织。
   - 突出来源证据，而不是普通列表。

4. 后端真实稳定性：
   - Postgres 持久化。
   - API token 强制。
   - runtime config 脱敏。
   - DeepSeek/Volc/AMap 代理 smoke。

### P2/P3：不阻塞当前真机闭环，但属于规划缺口

1. 独立小程序/智能音箱端。
2. 机构端 SaaS、租户和权益发放。
3. 专家转介/预约/支付。
4. 康养服务撮合。
5. 数字传记商品化交付。
6. 长期声纹训练和高质量数字分身生产链。

## 7. 建议调整后的阶段一验收定义

为了避免“docx 蓝图太大，工程永远验不完”，建议把阶段一验收定义拆成两层。

### 阶段一 A：当前工程应完成的真机 MVP

必须证明：

1. 失亲者可以完成最小建库：照片 + 文字 + 语音/截图 + 口头禅/性格。
2. 结构化知识库能生成真实人物、地点、事件、事实，并可追踪来源。
3. 时空信箱能延迟投递，正文不出端，回声有边界。
4. 数字人只引用授权记忆，不自由编造。
5. 危机表达触发停止扮演和干预入口。
6. 老人对话能生成脱敏关怀周报，子女不能看原始聊天。
7. 后端 metadata、鉴权、Postgres、第三方代理可用。

### 阶段一 B：完整规划的扩展验证

后续再证明：

1. 适老小程序/音箱端。
2. 真正声纹克隆训练链路。
3. 专家转介和机构服务。
4. 商业订阅/支付/服务撮合。
5. 长周期家庭账户生命周期闭环。

## 8. 结论

当前工程与《阶段一.docx》的核心精神并不冲突，但完成的是“底层能力和 A 端雏形”，不是完整的“数字哀伤辅导产品”。

最需要立刻纠正的不是继续加功能，而是把已有能力重新收束到三个验收主题：

1. 哀伤用户如何安全地倾诉和告别。
2. 记忆如何真实、可追踪、可授权地沉淀。
3. AI 如何克制地陪伴，而不是模拟复活。

如果按这个方向收束，现有工程可以继续推进；如果按 docx 全量蓝图验收，则还必须补 B 端适老终端、危机干预闭环、声纹模型生产链和机构商业后台。
